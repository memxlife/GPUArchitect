# Design Specification (Current)

## Version
v0.3

---

## 1. Current Objective

At this stage, the system aims to establish a **reliable and reproducible experimental loop** for GPU behavior exploration, and to extract the **first causal insights** about performance from structured benchmark sweeps.

The immediate goal is not full automation or optimal kernel generation, but:

> to validate that the system can **run controlled experiments, extract structured observations, and form causal interpretations of GPU behavior**.

---

## 2. Current System Architecture

The current system is a **minimal but complete vertical slice**:

### Benchmark Layer
- `benchmarks/compute.cu`
- `benchmarks/memory.cu`
- Parameterized CUDA kernels with CLI arguments

### Execution Layer
- `runner/run.py`
- Compiles CUDA code via `nvcc`
- Executes binary
- Optionally runs profiling (`ncu` ready)

### Parsing Layer
- `parser/parse_ncu.py`
- Extracts key-value outputs from stdout

### Model Layer
- `model/model.py`
- Stores latest experiment result
- Currently acts as a simple persistent store

### LLM Layer
- `llm/openai_client.py`
- `llm/interpret.py`
- Uses GPT-5.4 for analysis
- Used only for interpretation, not decision-making

### Entry Point
- `main.py`
- Orchestrates:
  - execution
  - parsing
  - storage
  - optional LLM analysis

---

## 3. Current Assumptions

The system currently operates under the following assumptions:

- The `compute_fma_like` kernel approximates **arithmetic intensity scaling**
- Increasing `inner_iters` increases compute work without significantly increasing memory traffic
- Benchmark output format is stable (`key=value`)
- Runtime measurements are stable enough with warmup + averaging
- LLM is used **only for interpretation**, not for updating the model
- Each experiment is assumed to represent a single dominant mechanism (not strictly enforced yet)

---

## 4. Experimental Evidence

A sweep over `inner_iters` was conducted:

- Fixed:
  - `n = 16777216`
  - `threads_per_block = 256`

- Swept:
  - `inner_iters = [1, 4, 16, 64, 256, 1024, 4096, 16384]`

### Observations:

- For `inner_iters ≤ 64`:
  - runtime ≈ 0.338 ms (flat)

- For `inner_iters ≥ 256`:
  - runtime increases significantly
  - scales approximately linearly with `inner_iters`

- At `inner_iters = 16384`:
  - checksum becomes `inf` (numerical overflow)

---

## 5. Current Knowledge (Causal Form)

### Claim
The compute benchmark exhibits two performance regimes.

### Evidence
- Flat runtime for small `inner_iters`
- Linear scaling for large `inner_iters`

### Why (Cause)
- Low arithmetic intensity → overhead-dominated regime
- High arithmetic intensity → compute throughput dominates

### How (Mechanism)
- Increasing `inner_iters` increases per-thread compute
- Memory traffic remains approximately constant
- Arithmetic intensity increases
- Kernel transitions from overhead/memory-limited → compute-limited

### So What (Implication)
- The system can detect **regime transitions**
- Compute-dominated region is suitable for:
  - occupancy studies
  - block size sensitivity
- Boundary region (~64–256) is most informative for refinement

### Uncertainty
- Exact transition point is unknown
- Role of memory in low-`inner_iters` regime is unclear
- No profiler counter validation yet

---

## 6. Known Issues / Failure Modes

- `compute.cu` becomes numerically unstable at large `inner_iters`
- Sweep output stores raw stdout instead of structured rows
- No validation for confounded experiments
- No error handling for unstable or invalid runs
- No profiler-based confirmation of bottlenecks
- LLM output is not validated or grounded in metrics

---

## 7. Design Gaps

The system currently lacks:

- Structured experiment registry (normalized data)
- Causal knowledge base representation
- Explicit uncertainty tracking
- Experiment selection / planning logic
- Mechanism isolation enforcement
- Integration of profiler metrics (e.g., SM utilization, memory throughput)
- Knowledge accumulation across runs (beyond latest result)

---

## 8. Design Decisions (This Iteration)

- Start with a **single benchmark family** (compute)
- Use **parameter sweep** instead of LLM-driven generation
- Keep system deterministic and simple
- Delay LangGraph / multi-agent orchestration
- Use LLM only for interpretation
- Focus on building first **causal insight** before scaling system complexity

---

## 10. Open Questions

- What is the exact transition boundary between regimes?
- What dominates runtime for small `inner_iters`?
- Is the kernel fully compute-bound at large `inner_iters`?
- How sensitive is performance to block size in compute regime?
- How does memory subsystem influence the early regime?
- What profiler metrics confirm the inferred behavior?
